# 03 — Data Structures (DB-less / In-Memory)

> This project is designed to run without a traditional relational database (DB-less). All catalog and geo data use static config files loaded into memory, and transactional write operations use append-only logs, in-memory caches, and direct API transmissions.

## 1) Core Data (Read-Only Static Files)

These files (JSON or CSV format) should be placed in `src/main/resources/data/`. They are generated by ETL scripts and loaded into Java `Map` or `List` data structures at application startup (e.g., via `@PostConstruct`).

### `counties.json` (or CSV)
Canonical geography metadata.
- **In-Memory Target**: `Map<String, County>` (Key: `state_abbr + "_" + county_slug`)
- **Fields**:
  - `state_abbr` 
  - `state_name`
  - `county_name`
  - `county_fips`
  - `epa_zone` (int, 1|2|3)
  - `slug` (text)
  - `page_path` (e.g. `/radon-mitigation-cost/oh/franklin-county`)

### `zip_primary_county.json` (or CSV)
Collapsed ZIP→primary county mapping for routing.
- **In-Memory Target**: `Map<String, String>` (Key: `zip`, Value: `county_fips`)

### `reference_sources.json`
Registry of citations shown in "Sources & Methodology".

## 2) Sellers and Lead Routing (Config)

### `sellers.json`
Since the network starts small, sellers can be defined in a local JSON configuration file instead of a DB table.
- **In-Memory Target**: `List<Seller>`
- **Fields**:
  - `id` (string UUID)
  - `display_name` (text)
  - `type` (contractor | network)
  - `states` (list of state_abbr)
  - `county_fips` (optional allowlist of counties they specifically service)
  - `email` (for direct email delivery)
  - `lead_post_url` (for webhook POSTs if using a CRM network)
  - `active` (boolean)

## 3) Transactional Data (Write-Only & Caching)

Since there is no DB, we handle Leads, Deduplication, and Audits via JVM memory and file streams.

### Deduplication (In-Memory Cache)
To prevent duplicate lead submissions and spam.
- **Mechanism**: Expiring Cache (e.g., Spring Cache + Caffeine) with a 24-hour TTL.
- **Key**: `sha256(lowercase(email) + '|' + normalized_phone + '|' + county_fips)`
- **Behavior**: If a form submission generates a key already in the cache, it is silently ignored or rejected.

### Consent & Audit Logging (Append-Only File)
For TCPA compliance, every explicit consent must be recorded permanently.
- **Mechanism**: Write to an append-only rolling log file (e.g., `logs/consent_audit.log` or `.csv`) or send asynchronously to an external system (e.g., Google Sheets).
- **Columns**: `timestamp, lead_id, ip_address, user_agent, consent_text, consent_version, selected_seller_ids, method`

### Lead Delivery Operations
There is no central `leads` DB table. The lead object lives in memory for the lifecycle of the request.
- **Action**: The system validates the form, checks dedupe, attempts transmission to the seller (email or API), and logs the outcome.
- **Mechanism**: Append result to `logs/lead_deliveries.log`.
- **Columns**: `timestamp, lead_id, seller_id, delivery_method, status (sent|failed)`

---

## 4) Security Requirements
- Ensure audit log files (`consent_audit.log`) are stored outside public web directories, regularly backed up, and access-restricted.
- Do not log raw PII (like plain phone numbers) in generic application debug logs. Limit PII only to the dedicated, secure audit files.
